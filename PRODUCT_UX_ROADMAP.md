# Smart TOC & Scroll — 产品与体验路线图

> 更新时间：2026-03-27
> 适用范围：产品规划、设计收敛、前端分阶段迭代

## 1. 当前项目是做什么的

Smart TOC & Scroll 是一个 Chrome 扩展，目标是在任意长文、博客、文档站点中提供轻量、低干扰的阅读导航能力。

当前产品已经具备两种阅读导航模型：

- `标准目录面板`
  - 面向文档站、技术博客、知识库
  - 用一个低侵扰浮标作为入口
  - 展开后显示完整 TOC、当前章节高亮和 `Top` 按钮
- `阅读进度目录（SSPAI）`
  - 面向内容型、沉浸式长文阅读
  - 用边缘线状目录表达章节位置
  - 默认只展示短横线与当前章节
  - 悬停后显示标题文字
  - 使用独立回顶按钮

从产品形态上看，这不是单一的“回到顶部插件”，而是一个面向阅读导航的轻量工具。

## 2. 当前功能与实现整理

### 2.1 已实现的核心能力

- 自动识别正文容器与标题
- 支持标准标题和部分自定义标题样式
- 当前章节高亮
- 点击目录项平滑跳转
- 平滑回顶
- 显示条件控制
  - 最少标题数
  - 滚动多少屏后显示
- 兼容策略
  - 禁用域名
  - 避开已有 TOC / 回顶控件
  - 强制显示
- SPA 页面更新与重建
- 浅色 / 深色主题自适应
- 键盘可达性
- 性能统计面板（默认关闭）

### 2.2 当前主要实现模块

- `manifest.json`
  - MV3 注入配置
- `catalog.js`
  - TOC 主流程、设置、内容识别、DOM 注入、交互、滚动高亮、性能采样
- `theme.js`
  - 主题判定和主题 class 应用
- `toc.css`
  - 默认目录面板和 `SSPAI` 阅读进度目录样式
- `themes.css`
  - 浅色 / 深色主题变量、预设变量
- `options.*`
  - 设置页 UI、保存、联动文案

### 2.3 当前优势

- 已经形成明确的阅读导航价值，不只是单点回顶
- 支持两种不同密度的导航模型
- 默认状态较轻，设置较完整
- 已开始关注低侵扰、键盘可达性和性能开销

### 2.4 当前主要短板

- 产品模型开始分叉，设置页与 README 之前没有把差异讲清楚
- `标准目录面板` 和 `SSPAI` 模式行为差异较大，用户学习成本偏高
- 避开已有控件仍依赖启发式检测，站点越多越容易出现误判 / 漏判
- `catalog.js` 已经承担过多职责，后续扩展成本在上升

## 3. 市场与最佳实践判断

从当前市面上的阅读辅助和滚动类插件形态看，用户最常接受的不是“更多按钮”，而是“更低干扰、更明确的单一价值”。

常见模式可以归纳为：

- `单纯回顶 / 回底按钮`
  - 强调简单、轻量、无数据采集
  - 优点是易理解
  - 缺点是产品价值单薄，难形成差异化
- `阅读进度条`
  - 强调位置感知
  - 低侵扰、学习成本低
  - 但导航能力有限
- `完整 TOC / Outline 面板`
  - 适合标题结构丰富的文档
  - 导航效率高
  - 但容易侵扰页面视觉

因此，当前项目真正有竞争力的方向不是和“回顶按钮插件”正面同质化，而是强化：

- 低干扰的阅读导航
- 同时覆盖“完整目录”和“位置感知”两类需求
- 站点兼容稳定
- 设置清晰但不复杂

## 4. 产品判断：当前项目的优劣

### 4.1 当前做得好的地方

- 产品价值比单纯回顶更完整
- 已经有明显差异化的 `SSPAI` 阅读目录方向
- 对性能和侵扰感是有意识收敛的
- 具备作为 Chrome Store 长期维护产品的基础

### 4.2 当前不够强的地方

- 缺少“产品语义收敛”
  - 用户不清楚自己是在选主题，还是在选一种阅读导航模型
- 缺少“兼容性可信度”
  - 避让机制一旦失效，会直接伤害产品信任
- 缺少“新手理解路径”
  - 尤其是 `SSPAI` 模式，优雅但不够自解释
- 缺少“渐进增强”
  - 不同模式间的行为切换有时像换了一个产品

## 5. 改进原则

后续迭代统一遵循四个原则：

- `低侵扰优先`
  - 默认状态尽量退后，不破坏宿主页面阅读节奏
- `阅读导航优先`
  - 强化“位置感知 + 快速跳转”，不要退化成普通按钮集合
- `行为清晰优先`
  - 同一模式下入口职责稳定，不让用户猜
- `性能与稳定性优先`
  - 尽量少轮询、少重建、少布局抖动

## 6. 分阶段改进方案

### Phase 1：收敛产品语义与高频阅读体验

目标：

- 把“主题预设”明确成“阅读导航样式”
- 补齐文档和设置页说明
- 修正高频阅读中的体验细节

已纳入本阶段的方向：

- 设置页文案改成“标准目录面板” / “阅读进度目录（SSPAI）”
- `press` 降为高级模式文案
- 长目录中活动章节自动滚入可视区
- 点击目录项时立即同步高亮并保持该项可见

验证重点：

- 用户能否快速理解两种模式差异
- 长目录阅读时，高亮项是否始终容易被看到

### Phase 2：重构已有控件避让

目标：

- 降低重复注入
- 降低误跳过

策略：

- 通用启发式规则 + 站点特征规则双层检测
- 内部记录跳过原因，方便后续兼容性诊断

当前已开始落地：

- 将“是否跳过注入”升级为带类型与来源的决策对象
- 为 `sspai.com` 增加首个站点特征检测器
- 在页面上记录 `data-smart-toc-skip-reason` 与 `window.__SMART_TOC_LAST_SKIP__` 便于调试
- 增加 `window.__SMART_TOC_WIDGET_DIAGNOSTICS__` 作为兼容性快照，便于后续扩站点规则
- 对正文主内容里的内联目录做单独识别，避免被误判成侧栏 TOC
- 增加 `window.__SMART_TOC_INSPECT_WIDGETS__()`，可主动刷新兼容性检测快照

验证重点：

- 少数派这类边缘目录页面
- 有正文内小目录但不应跳过的页面
- 仅有回顶按钮的页面

### Phase 3：拆分布局模式与交互控制器

目标：

- 降低 `catalog.js` 的条件分支复杂度
- 让默认面板和 `SSPAI` rail 变成两套可维护的 renderer/controller

策略：

- 抽离 layout renderer
- 抽离 interaction controller
- 保留统一的标题、滚动和设置 orchestration

当前已开始落地：

- `标准目录面板` 与 `SSPAI` 阅读目录已拆成独立的 UI 创建函数
- 两种模式的交互绑定已拆成独立 controller 入口
- 主流程开始只负责分派，不再在同一个函数里堆大量 preset 分支
- 目录项渲染开始抽成独立 renderer 逻辑，不再直接写在 `updateTOC()` 内部

### Phase 4：优化滚动高亮与目录更新性能

目标：

- 降低长文页滚动时的布局读取成本
- 减少 DOM 重建

策略：

- `IntersectionObserver` 优先
- 标题签名缓存
- 结构不变时不重建整棵 TOC

当前已开始落地：

- 已加入标题签名缓存
- 标题结构不变时，`updateTOC()` 不再重建整棵目录，只同步计数与当前高亮
- 当前章节高亮开始由 `IntersectionObserver` 优先驱动，滚动时仅在不支持观察器时回退到位置遍历
- 标题结构不变时，不再重复重建标题观察器，减少动态页面上的额外监听开销
- DOM 观察器会忽略扩展自身节点的插入与更新，避免插件 UI 反向触发无意义重建
- 已补充 Astro / View Transitions 类站点在页面交换、恢复与 BFCache 场景下的重建兜底

### Phase 5：完善品牌感与 Chrome Store 竞争力

目标：

- 形成更统一的产品叙事
- 提升商店转化和长期留存

策略：

- 更新商店描述与截图
- 做两种导航样式的场景展示
- 准备兼容性说明与隐私承诺

当前已开始落地：

- 商店定位从“回顶按钮”收敛到“轻量阅读导航扩展”
- 已整理 Chrome Web Store 文案草稿、截图脚本和隐私 / 信任说明
- 扩展 manifest 描述已同步为更贴近阅读导航定位的版本

## 7. 当前推荐的产品定位

最建议的产品定义是：

> 一个面向长文和文档站点的轻量阅读导航扩展，兼顾“完整目录导航”和“低侵扰进度目录”两种阅读方式。

这比“Scroll to Top 插件”更有市场竞争力，也更符合当前代码和设计的真实演进方向。

## 8. 当前阶段测试清单

### 核心行为

- 标准目录面板是否正确显示、展开、跳转、回顶
- `SSPAI` 阅读进度目录是否能正确高亮当前章节
- hover 展开后，标题是否出现在短横线外侧
- 左右位置切换后，短横线是否始终靠正文一侧

### 兼容性

- 页面已有 TOC 时是否正确避让
- 页面已有回顶按钮时是否正确避让
- 强制显示开启后是否忽略避让

### 性能

- 长文页面滚动时是否有明显卡顿
- 标题很多时当前高亮项是否仍能及时跟随
- 默认状态下是否保持低侵扰
